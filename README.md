# CS-360-Mobile-Architect-Programming

# Portfolio Reflection

# Briefly summarize the requirements and goals of the app you developed. What user needs was this app designed to address?
The WeightTracker app was designed to address the need for a simple, private tool to track body weight over time. The core requirements included user account creation and login, the ability to log daily weights, setting a goal weight, and viewing weight history. Users needed a way to consistently monitor their progress toward a weight goal without relying on paper logs or complex fitness apps. The app was designed to be straightforward open it, log your weight, and see how you're trending.

# What screens and features were necessary to support user needs and produce a user-centered UI for the app? How did your UI designs keep users in mind? Why were your designs successful?
The app required four primary screens: a Login screen for authentication, an Add Weight screen for daily entries, a Goal Weight screen for setting targets, and a Weight History screen showing logged entries in a scrollable list. The UI kept users in mind by minimizing the number of taps required to complete any action logging a weight takes just one screen with a date field, weight field, and submit button. The designs were successful because they followed Material Design conventions that Android users already understand, used clear labels and input hints, and maintained a consistent visual hierarchy across all screens. 

# How did you approach the process of coding your app? What techniques or strategies did you use? How could those techniques or strategies be applied in the future?
I approached coding by first establishing the data layer (DatabaseHelper with SQLite), then building each Activity to handle its specific screen logic. I used a separation of concerns pattern where the DatabaseHelper class encapsulates all database operations like create user, add weight, get weights, set goal and each Activity only handles UI interaction and delegates data operations to the helper. I also used Android's Activity lifecycle methods to ensure data loads correctly when screens are opened or returned to. These strategies building the data layer first, separating UI from data logic, and following platform lifecycle patterns apply to any mobile or client-server application development.

# How did you test to ensure your code was functional? Why is this process important, and what did it reveal?
I tested the app using the Android Emulator to simulate real device behavior, manually walking through each user flow: registration, login, adding weights, viewing history, and setting a goal. I also used Android Studio's instrumented test framework. Testing was important because it revealed edge cases like handling empty input fields, duplicate user registrations, and ensuring the weight list updated correctly after adding a new entry. The process confirmed that the database operations worked correctly across app restarts, proving that SQLite persistence was functioning as intended.

# Consider the full app design and development process from initial planning to finalization. Where did you have to innovate to overcome a challenge?
The biggest challenge was designing the weight history display to be both informative and performant. I had to innovate by creating a custom row layout item_weight_row.xml and binding it dynamically to the weight history list rather than using a static layout. This approach allowed the list to scale to any number of entries without hardcoding UI elements. I also had to work through the challenge of passing data between Activities correctly using Android's Intent system, ensuring that the goal weight set on one screen was accurately reflected and persisted for comparison on other screens.

# In what specific component of your mobile app were you particularly successful in demonstrating your knowledge, skills, and experience?
I was most successful in the DatabaseHelper component. It demonstrates solid understanding of SQLite schema design, parameterized queries for security, proper use of `SQLiteOpenHelper` for database versioning, and clean method interfaces that the Activities can call without needing to know SQL syntax. The helper handles user table creation, weight entries with timestamps, and goal weight storage all through well-named methods that accept simple parameters and return usable results.
